# جمع‌آوری نیازهای رابط کاربری و تعریف نقش‌ها — فاز ۷ (مرحله ۱)

مسیر پروژه: `multi-messaging-platform/`

## ۱) جریان‌های اصلی کاربر (User Flows)

### ۱.۱ ورود/احراز هویت (Login)
- **پیش‌نیاز**: کاربر در سیستم تعریف شده باشد (Admin/Operator/Viewer).
- **ورودی‌ها**: نام کاربری، رمز عبور.
- **خروجی‌ها**:
  - توکن/سشن معتبر برای دسترسی به داشبورد
  - نمایش خطاهای رایج: نام کاربری/رمز اشتباه، کاربر غیرفعال، خطای ارتباط با سرویس
- **رفتارهای UI مهم**:
  - نمایش وضعیت بارگذاری (loading) هنگام ارسال فرم
  - پیام خطای قابل فهم فارسی (RTL)
  - دکمه خروج از حساب و انقضای سشن

### ۱.۲ افزودن و مدیریت اکانت‌ها (Accounts) برای پلتفرم‌ها
پلتفرم‌ها: **واتساپ، تلگرام، بله، روبیکا**

- **هدف**: تعریف کانال/اکانت برای ارسال پیام + مشاهده وضعیت اتصال
- **عملیات کلیدی**:
  - افزودن اکانت (فرم با فیلدهای لازم برای هر پلتفرم)
  - ویرایش اکانت
  - فعال/غیرفعال کردن یا حذف
  - تست اتصال / مشاهده آخرین خطا
- **خروجی‌ها**:
  - لیست اکانت‌ها با وضعیت (Connected / Error / Disabled)
  - ثبت رویدادهای حساس (برای نقش‌های مجاز) در Audit

### ۱.۳ بارگذاری مخاطبان از Excel/CSV + پیش‌نمایش
- **هدف**: وارد کردن لیست مخاطبان و آماده‌سازی برای کمپین
- **مراحل پیشنهادی UI**:
  - انتخاب فایل (Drag & Drop + انتخاب دستی)
  - تشخیص فرمت (CSV/XLSX) و Encoding
  - نمایش پیش‌نمایش ۲۰–۵۰ سطر
  - **مپینگ ستون‌ها** (مثلاً: شماره/شناسه، نام، متغیرهای قالب پیام)
  - اعتبارسنجی: الزامی‌ها، تکراری‌ها، مقادیر نامعتبر
  - ذخیره لیست با نام/توضیح (اختیاری)
- **خروجی‌ها**:
  - لیست‌های ذخیره‌شده قابل انتخاب در ساخت کمپین
  - گزارش خطاهای ردیفی با امکان دانلود

### ۱.۴ ایجاد کمپین پیام‌رسانی (Campaign Create)
- **هدف**: ساخت یک کمپین با انتخاب کانال، متن/قالب، و سیاست سرعت
- **الگوی پیشنهادی**: Wizard چندمرحله‌ای
  - انتخاب پلتفرم/اکانت
  - انتخاب لیست مخاطبان
  - متن پیام:
    - دستی (Template با متغیرها)
    - یا تولید/بازنویسی با GPT (با prompt و کنترل‌های ساده)
  - تنظیمات سرعت/Rate limit (مثلاً پیام بر ثانیه/دقیقه، یا batch size)
  - زمان‌بندی (الزامی/اختیاری): فوری یا زمان‌دار
  - خلاصه نهایی + ایجاد کمپین
- **خروجی‌ها**:
  - ایجاد کمپین با وضعیت اولیه (Draft/Ready/Queued)
  - لینک به صفحه پایش کمپین

### ۱.۵ مدیریت و پایش کمپین‌ها (Campaign Monitor & Control)
- **هدف**: کنترل اجرای کمپین + مشاهده وضعیت ارسال‌ها
- **عملیات کلیدی**:
  - شروع/توقف/مکث (اگر پشتیبانی شود)
  - مشاهده پیشرفت (percent + شمارنده‌ها)
  - مشاهده throughput (نمودار زمانی)
  - فیلتر وضعیت پیام‌ها (Queued/Sent/Failed/Rate-limited/Skipped)
  - دانلود گزارش خروجی (CSV/XLSX)
  - تغییر برخی تنظیمات مجاز (مثلاً سرعت) با کنترل نقش
- **خروجی‌ها**:
  - گزارش قابل استناد برای عملیات
  - هشدارهای قابل مشاهده در UI (خطاهای اکانت، نرخ محدودیت، قطعی Redis/DB)

### ۱.۶ گزارش‌ها و لاگ‌ها (Reports, Audit Logs, Message Logs)
- **هدف**: ردیابی فعالیت‌ها و عیب‌یابی/گزارش‌گیری
- **زیر بخش‌ها**:
  - **Message Logs**: لاگ ارسال‌ها (برای عملیات روزانه)
  - **Audit Logs**: اقدامات حساس (مدیریت اکانت‌ها، تغییر تنظیمات، تغییر نقش‌ها، …)
- **نیازهای UI**:
  - فیلتر زمان/کمپین/اکانت/کاربر/وضعیت
  - Pagination
  - Export
  - نمایش جزئیات یک رکورد (Modal/Drawer)

## ۲) تعریف نقش‌ها و سطح دسترسی (RBAC)

### ۲.۱ تعریف نقش‌ها
- **Admin**: دسترسی کامل (مدیریت اکانت‌ها، ساخت/ویرایش کمپین‌ها، کنترل اجرا، تنظیم سرعت، مشاهده گزارش‌ها و **Audit Logs**، تنظیمات حساس)
- **Operator**: عملیات اجرا (بارگذاری مخاطبان، ساخت کمپین، شروع/توقف کمپین، مشاهده گزارش‌های عملیاتی) بدون تنظیمات حساس و بدون Audit
- **Viewer**: فقط مشاهده وضعیت/گزارش‌ها (Read-only) بدون ایجاد/ویرایش/کنترل اجرا

### ۲.۲ ماتریس مجوزها (بدون تداخل)

#### Authentication / Session
- **Login**: Admin/Operator/Viewer ✅
- **Logout**: Admin/Operator/Viewer ✅

#### Accounts
- **View accounts list/status**: Admin ✅ | Operator ✅ | Viewer ✅
- **Add account**: Admin ✅ | Operator ❌ | Viewer ❌
- **Edit account**: Admin ✅ | Operator ❌ | Viewer ❌
- **Disable/Delete account**: Admin ✅ | Operator ❌ | Viewer ❌
- **Test connection**: Admin ✅ | Operator ✅ (اختیاری) | Viewer ❌

#### Contacts / Audience lists
- **View lists**: Admin ✅ | Operator ✅ | Viewer ✅ (فقط مشاهده)
- **Upload new list (CSV/XLSX)**: Admin ✅ | Operator ✅ | Viewer ❌
- **Edit list metadata / delete list**: Admin ✅ | Operator ✅ (محدود) | Viewer ❌

#### Campaigns
- **View campaigns & status**: Admin ✅ | Operator ✅ | Viewer ✅
- **Create campaign**: Admin ✅ | Operator ✅ | Viewer ❌
- **Edit campaign (pre-run)**: Admin ✅ | Operator ✅ (محدود) | Viewer ❌
- **Start/Stop campaign**: Admin ✅ | Operator ✅ | Viewer ❌
- **Change speed/rate limit settings**: Admin ✅ | Operator ❌ | Viewer ❌
- **Download campaign report**: Admin ✅ | Operator ✅ | Viewer ✅ (Read-only export)

#### Logs / Reports
- **Message Logs (view/export)**: Admin ✅ | Operator ✅ | Viewer ✅ (view + export محدود)
- **Audit Logs (view/export)**: Admin ✅ | Operator ❌ | Viewer ❌

#### Settings / Sensitive areas
- **System settings (secrets, rate caps, RBAC changes)**: Admin ✅ | Operator ❌ | Viewer ❌

> نکته: هر آیتمی که «تغییر وضعیت» ایجاد می‌کند (Create/Update/Delete/Start/Stop/Change-speed) باید در UI با کنترل نقش، و در سمت سرور با enforce مجوز همراه باشد.

## ۴) پرسش‌های باز برای تکمیل الزامات UI
- **اعلان‌ها**:
  - آیا اعلان‌های درون‌برنامه‌ای کافی است یا SMS/Email هم لازم است؟
  - اعلان‌ها برای چه رویدادهایی؟ (پایان کمپین، خطای اکانت، نرخ محدودیت شدید، خطای DB/Redis)
- **گزارش‌های داشبورد**:
  - KPIهای ضروری دقیقاً کدام‌اند؟ (روزانه/هفتگی/برحسب کمپین/کانال)
  - آیا نیاز به export زمان‌بندی‌شده داریم؟
- **کمپین‌ها**:
  - آیا «مکث/ادامه» لازم است یا فقط start/stop؟
  - آیا امکان ویرایش متن پس از شروع کمپین مجاز است؟
  - سیاست re-try برای پیام‌های Failed در UI قابل کنترل باشد؟
- **مخاطبان**:
  - فرمت شناسه مخاطب برای هر پلتفرم چیست؟ (شماره تلفن/یوزرنیم/آیدی عددی)
  - آیا نیاز به dedupe cross-list داریم؟
  - آیا فیلدهای سفارشی (custom attributes) لازم است؟
- **دسترسی‌ها**:
  - Viewer اجازه دانلود گزارش داشته باشد یا فقط مشاهده؟
  - Operator اجازه تست اتصال اکانت داشته باشد؟
- **چند-سازمانی/چند-tenant**:
  - آیا کاربران چند workspace دارند یا فقط یک سازمان؟

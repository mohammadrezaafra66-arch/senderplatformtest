## 0. توضیح مهم درباره نام «Afra Kala / افراکالا»

در کد این مخزن، نام رسمی سرویس **«Multi Messaging Platform»** یا **«Sender Platform»** است (`core_engine/main.py`: `title="Multi Messaging Platform"`).  
نام **افراکالا** به‌عنوان **دامنه کسب‌وکار B2B** در محتوا دیده می‌شود:

- `storage/knowledge_base/default_sales_kb.md` — «پایگاه دانش فروش افراکالا»
- `core_engine/services/gpt_orchestrator.py` — پرامپت «نویسنده پیام فروش B2B برای افراکالا»
- `core_engine/services/pricing_scraper.py` — «اسکرپر HTML برای قیمت‌های افراکالا»
- `frontend/README.md` — اشاره به کانتینر جداگانه `afrakala-local-web` روی پورت 3000

**این بخش در کد فعلی یافت نشد:** یک اپلیکیشن monolithic جداگانه به نام Afra Kala داخل همین repo. کانتینرهای `afrakala-local-*` سرویس‌های **خارج از این مخزن** هستند که روی همان ماشین dev اجرا می‌شوند.

## 2. شرح ماژول‌ها (بر اساس محتوای واقعی فایل‌ها)

### 2.1 `core_engine/main.py` — نقطه ورود API

- FastAPI app با CORS برای پورت‌های 5173، 3000، 3010.
- Routerها: auth، accounts، campaigns، audit، controls، dashboard، imports، metrics، debug routes.
- Endpointهای مستقیم روی app: `/health`, `/debug/gpt/*`, `/debug/pricing/*`, `/debug/render/*`, `/debug/safety-status`, `/debug/celery/add`.
- در startup، `SESSION_SECRET` از طریق `get_settings()` اعتبارسنجی می‌شود.
- وابستگی: تمام service layer و OpenAI/pricing/GPT orchestrator.

### 2.2 `core_engine/config.py` — تنظیمات

- Pydantic Settings از `.env`: DATABASE_URL، REDIS_URL، OPENAI (مدل پیش‌فرض `gpt-4o-mini`)، pricing scraper، safety gates Phase 4–9.
- Gates پیش‌فرض **خاموش**: `REAL_QUEUE_PUSH_ENABLED`, `REAL_MESSAGE_SENDING_ENABLED`, `WORKER_EXECUTION_ENABLED`, `CHANNEL_CONNECTORS_ENABLED`, `OPS_LIVE_SEND_API_ENABLED`.
- `WHATSAPP_OPS_SEND_VIA_WORKER_QUEUE` برای route کردن live send UI به Redis worker (Windows host).
- `SESSION_SECRET` اجباری و باید Fernet-valid باشد.

### 2.3 `core_engine/models.py` — مدل داده

ORM SQLAlchemy شامل:

- **Account** — پلتفرم (whatsapp/telegram/bale/rubika)، phone، status
- **Campaign**, **Contact**, **CampaignRecipient**, **Message**, **MessageAttempt**
- **StagedQueueItem** — صف staging Phase 4
- **ChannelSession** — session رمزنگاری‌شده (Fernet) برای bot tokens / WhatsApp Web metadata
- **ImportBatch**, **ProductSnapshot**, **RenderedMessage**
- **User**, **AuditLog** — جدول users migrate شده ولی auth از DB استفاده **نمی‌کند**
- Enumهای وضعیت send/render/consent

### 2.4 `core_engine/api/auth.py`

- `POST /auth/token` — OAuth2 password flow
- `GET /auth/me`
- کاربران در **`fake_users_db` in-memory**: admin/admin123, operator/operator123, viewer/viewer123
- JWT با `SECRET_KEY`؛ role در payload
- `require_roles()` decorator برای RBAC

### 2.5 `core_engine/api/accounts.py`

(بر اساس imports و tests — endpoints اصلی):

- CRUD اکانت‌ها
- `POST /{id}/test-connection`
- WhatsApp Web: status، register session، pool status
- API token session برای Bale/Telegram/Rubika
- `GET /operational-send/capabilities`, `GET /{id}/operational-send/preflight`
- `POST /{id}/send-test` — dry-run و live send با gateهای env
- deploy readiness

### 2.6 `core_engine/services/gpt_orchestrator.py`

- `preview_personalized_message()` و `generate_personalized_message()`
- پرامپت فارسی B2B افراکالا + knowledge base + product snapshot
- فراخوانی واقعی `client.chat.completions.create` با JSON schema
- وابسته به `openai_client.py` — اگر API key placeholder باشد، خطا می‌دهد

### 2.7 `core_engine/services/message_render.py`

- `dry_run_message_render()` — render بدون send
- `build_mock_personalized_message()` — متن mock فارسی با نام محصولات افراکالا
- اگر `force_mock_output=True` → mock؛ در غیر این صورت GPT واقعی

### 2.8 `core_engine/services/phase4_prepare.py`

- آماده‌سازی پیام کمپین → **فقط DB staging**
- `build_phase4_mock_final_text()` — متن mock صریح
- `render_mode="mock"` در staging

### 2.9 `core_engine/services/queue_bridge.py`

- Claim کردن `StagedQueueItem` با status READY
- Push JSON به Redis `queue:{platform}:{account_id}` **فقط اگر** `REAL_QUEUE_PUSH_ENABLED=true`
- consent gate قبل از push

### 2.10 `core_engine/services/operational_send.py`

- Phase 9.2: send-test تک‌پیام با preflight checks
- `build_test_worker_payload()` — payload aisled برای ops-test
- اگر `WHATSAPP_OPS_SEND_VIA_WORKER_QUEUE=true` → enqueue به Redis (نه Playwright مستقیم در API container)
- preflight: env gates، account status، WhatsApp Web linked، worker pool heartbeat

### 2.11 `core_engine/services/whatsapp_web_session.py`

- ذخیره/بارگذاری metadata session WhatsApp Web در `ChannelSession`
- `resolve_whatsapp_runtime_profile_dir()` — روی Windows اول LOCALAPPDATA را ترجیح می‌دهد
- `build_whatsapp_web_status()` — linked، needs_qr، profile_exists

### 2.12 `core_engine/tasks.py` — Celery

- `celery_app` روی Redis broker/backend
- Tasks: `add_numbers`, `send_message_task`, `push_ready_staged_items`
- Beat هر 5 ثانیه staged items را به queue push می‌کند

### 2.13 `workers/delivery.py`

- مسیر dispatch: `DRY_RUN` → synthetic ID | `SHADOW_MODE` → shadow | live connectors
- Live نیاز به `REAL_MESSAGE_SENDING_ENABLED` + `CHANNEL_CONNECTORS_ENABLED`
- Platform mapping: bale, telegram, rubika, whatsapp (web/cloud)
- Unknown platform → `placeholder_not_implemented`

### 2.14 `workers/connectors/whatsapp_web.py` + `playwright_sender.py`

- Playwright + persistent Chrome profile
- `send_whatsapp_web_message()`: navigate به `/send?phone=`, type text، click send، verify outgoing bubble
- `probe_whatsapp_web_session()` برای logged_in check
- **وضعیت عملیاتی:** روی Windows host با `whatsapp_worker_pool_windows.ps1` تست شده؛ مشکل false-positive «delivered» در حال اصلاح بود

### 2.15 `workers/connectors/whatsapp.py`

- WhatsApp **Cloud API** (Meta Graph) via httpx
- Token از `ChannelSession` decrypt
- `resolve_whatsapp_recipient()` — normalize E.164

### 2.16 `workers/connectors/telegram.py`, `bale.py`, `rubika.py`

- HTTP POST به Bot API هر پلتفرم
- Token از encrypted session
- Retryable vs permanent errors

### 2.17 `workers/whatsapp_pool_worker.py`

- Multi-account round-robin
- Browser lock (process + Redis distributed)
- Rate limit، hourly cap، retry، heartbeat در Redis
